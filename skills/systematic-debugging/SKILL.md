# 系统化调试

## 概述

随机修复浪费时间并引入新的 bug。快速补丁掩盖了底层问题。

**核心原则：** 在尝试修复之前，务必找到根本原因。修复表面症状就是失败。

**违反流程的字面意思就是违反调试的精神。**

## 铁律

```
没有根因调查，就不进行修复
```

如果你没有完成第一阶段，你就不能提出修复方案。

## 何时使用

用于任何技术问题：
- 测试失败
- 生产环境 bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**尤其在以下情况使用：**
- 时间紧迫（紧急情况使人倾向于猜测）
- "一个快速修复"看起来很明显
- 你已经尝试了多次修复
- 上一次修复没有起作用
- 你没有完全理解问题

**不要在以下情况跳过：**
- 问题看起来很简单（简单的 bug 也有根本原因）
- 你很着急（赶时间反而保证返工）
- 领导要求立刻修好（系统化方法比反复试错更快）

## 四个阶段

你必须完成每个阶段后才能进入下一个。

### 第一阶段：根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们通常包含确切的解决方案
   - 完整阅读堆栈跟踪
   - 记录行号、文件路径、错误代码

2. **稳定复现**
   - 能否可靠地触发问题？
   - 确切的步骤是什么？
   - 每次都能复现吗？
   - 如果无法复现 → 收集更多数据，不要猜测

3. **检查最近的变更**
   - 什么变更可能导致了这个问题？
   - Git diff、最近的提交
   - 新的依赖、配置变更
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统有多个组件时（CI → 构建 → 签名，API → 服务 → 数据库）：**

   **在提出修复方案之前，添加诊断埋点：**
   ```
   对每个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置的传播
     - 检查每一层的状态

   运行一次以收集证据，显示在哪里断裂
   然后分析证据以定位故障组件
   然后调查那个特定的组件
   ```

   **示例（多层系统）：**
   ```bash
   # 第一层：工作流
   echo "=== 工作流中可用的密钥: ==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # 第二层：构建脚本
   echo "=== 构建脚本中的环境变量: ==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # 第三层：签名脚本
   echo "=== Keychain 状态: ==="
   security list-keychains
   security find-identity -v

   # 第四层：实际签名
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这揭示了：** 哪一层失败（密钥 → 工作流 ✓，工作流 → 构建 ✗）

5. **追踪数据流**

   **当错误发生在调用栈深处时：**

   参见本目录下的 `root-cause-tracing.md` 了解完整的反向追踪技术。

   **快速版本：**
   - 错误的值从哪里来？
   - 什么代码用错误的值调用了这里？
   - 持续向上追踪直到找到源头
   - 在源头修复，而不是在症状处

### 第二阶段：模式分析

**在修复之前找到模式：**

1. **找到可工作的示例**
   - 在同一代码库中找到类似的、可正常工作的代码
   - 什么类似的东西是正常工作的？

2. **与参考实现对比**
   - 如果在实现某个模式，完整阅读参考实现
   - 不要略读——逐行阅读
   - 在应用之前完全理解该模式

3. **识别差异**
   - 正常工作的和出问题的之间有什么不同？
   - 列出每一个差异，不论多小
   - 不要假设"那不重要"

4. **理解依赖关系**
   - 这个组件还需要什么？
   - 什么设置、配置、环境？
   - 它做了什么假设？

### 第三阶段：假设与测试

**科学方法：**

1. **形成单一假设**
   - 清晰陈述："我认为 X 是根本原因，因为 Y"
   - 写下来
   - 要具体，不要模糊

2. **最小化测试**
   - 做出最小的可能变更来测试假设
   - 每次只改一个变量
   - 不要同时修复多个问题

3. **继续之前先验证**
   - 有效了？是 → 第四阶段
   - 无效？形成新的假设
   - 不要在上面叠加更多修复

4. **当你不知道时**
   - 说"我不理解 X"
   - 不要假装知道
   - 寻求帮助
   - 做更多研究

### 第四阶段：实施

**修复根本原因，而非表面症状：**

1. **创建失败的测试用例**
   - 最简单的复现方式
   - 尽可能使用自动化测试
   - 如果没有测试框架就用一次性测试脚本
   - 修复之前必须先有测试
   - 使用 `zonedev:test-driven-development` skill 来编写正确的失败测试

2. **实施单一修复**
   - 解决已识别的根本原因
   - 每次只做一个变更
   - 没有"顺便改进一下"
   - 不捆绑重构

3. **验证修复**
   - 测试现在通过了吗？
   - 其他测试没有被破坏吧？
   - 问题真正解决了吗？

4. **如果修复不起作用**
   - STOP
   - 计数：你已经尝试了多少次修复？
   - 如果 < 3 次：回到第一阶段，用新信息重新分析
   - **如果 >= 3 次：STOP 并质疑架构（见下面第 5 步）**
   - 不要在没有架构讨论的情况下尝试第 4 次修复

5. **如果 3 次以上修复都失败了：质疑架构**

   **表明架构问题的模式：**
   - 每次修复都暴露出新的共享状态/耦合/不同位置的问题
   - 修复需要"大规模重构"才能实施
   - 每次修复都在其他地方产生新的症状

   **STOP 并质疑根本问题：**
   - 这个模式从根本上是合理的吗？
   - 我们是否因为惯性而在坚持？
   - 应该重构架构还是继续修复症状？

   **在尝试更多修复之前与你的用户讨论**

   这不是假设失败——这是架构不对。

## 危险信号 - STOP 并遵循流程

如果你发现自己在想：
- "先快速修一下，之后再调查"
- "试试改一下 X 看看能不能行"
- "加多个改动，跑跑测试"
- "跳过测试，我手动验证一下"
- "大概是 X 的问题，我修一下"
- "我没有完全理解但这个可能有用"
- "模式说 X 但我换个方式适配一下"
- "主要问题有这些：[在没有调查的情况下列出修复方案]"
- 在追踪数据流之前就提出解决方案
- **"再试一次修复"（已经尝试了 2 次以上）**
- **每次修复都在不同的地方暴露新问题**

**以上所有情况都意味着：STOP。回到第一阶段。**

**如果 3 次以上修复都失败了：** 质疑架构（见第四阶段第 5 步）

## 你的用户发出的信号说明你做错了

**注意这些纠正：**
- "那不是没发生吗？" —— 你在没有验证的情况下做了假设
- "它会告诉我们……吗？" —— 你本应添加证据收集
- "别猜了" —— 你在没有理解的情况下提出修复
- "深入思考一下" —— 质疑根本问题，而不只是症状
- "我们卡住了？"（沮丧的语气） —— 你的方法不起作用

**当你看到这些信号时：** STOP。回到第一阶段。

## 常见的自我合理化

| 借口 | 现实 |
|--------|---------|
| "问题很简单，不需要走流程" | 简单的问题也有根本原因。流程处理简单 bug 很快。 |
| "紧急情况，没时间走流程" | 系统化调试比反复猜测试错更快。 |
| "先试这个，然后再调查" | 第一次修复定下了模式。从一开始就做对。 |
| "确认修复有效后我再写测试" | 没有测试的修复不可靠。先写测试来证明。 |
| "一次修多个问题节省时间" | 无法隔离哪个起了作用。会引入新 bug。 |
| "参考实现太长了，我改编一下这个模式" | 不完整的理解必然导致 bug。完整阅读。 |
| "我看到问题了，让我来修" | 看到症状 不等于 理解根本原因。 |
| "再试一次修复"（已经失败 2 次以上） | 3 次以上失败 = 架构问题。质疑模式，不要继续修。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|-------|---------------|------------------|
| **1. 根因调查** | 阅读错误、复现、检查变更、收集证据 | 理解了什么发生了以及为什么 |
| **2. 模式分析** | 找到可工作的示例、对比 | 识别出差异 |
| **3. 假设测试** | 形成理论、最小化测试 | 假设被确认或形成新假设 |
| **4. 实施修复** | 创建测试、修复、验证 | Bug 已解决，测试通过 |

## 当流程揭示"没有根本原因"时

如果系统化调查显示问题确实是环境相关的、时序依赖的或外部因素导致的：

1. 你已经完成了流程
2. 记录你调查了什么
3. 实施适当的处理（重试、超时、错误提示）
4. 添加监控/日志以便后续调查

**但是：** 95% 的"没有根本原因"情况实际上是调查不充分。

## 辅助技术

这些技术是系统化调试的一部分，存放在本目录中：

- **`root-cause-tracing.md`** - 通过调用栈反向追踪 bug 以找到原始触发点
- **`defense-in-depth.md`** - 找到根本原因后在多个层添加校验
- **`condition-based-waiting.md`** - 用条件轮询替代随意的超时等待

**相关 skill：**
- **zonedev:test-driven-development** - 用于创建失败的测试用例（第四阶段，第 1 步）
- **zonedev:verification-before-completion** - 在宣称成功之前验证修复是否有效

## 实际效果

来自调试实践的数据：
- 系统化方法：15-30 分钟完成修复
- 随机修复方法：2-3 小时的反复试错
- 一次修复成功率：95% vs 40%
- 引入新 bug：接近零 vs 经常发生
