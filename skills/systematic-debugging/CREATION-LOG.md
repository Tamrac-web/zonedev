# 创建日志：系统化调试 Skill

提取、结构化和强化关键 skill 的参考示例。

## 源材料

从 `~/.claude/CLAUDE.md` 提取的调试框架：
- 4 阶段系统化流程（调查 → 模式分析 → 假设 → 实施）
- 核心准则：始终找到根因，绝不修复表象
- 规则设计目标：抵御时间压力和自我合理化

## 提取决策

**纳入的内容：**
- 完整的 4 阶段框架及所有规则
- 反捷径规则（"绝不修复表象"、"停下来重新分析"）
- 抗压措辞（"即使更快也不行"、"即使我看起来很着急"）
- 每个阶段的具体步骤

**排除的内容：**
- 项目特定的上下文
- 同一规则的重复表述
- 叙述性解释（已浓缩为原则）

## 遵循 skill-creation/SKILL.md 的结构

1. **丰富的 when_to_use** - 包含症状和反模式
2. **类型：technique** - 带有步骤的具体流程
3. **关键词** - "root cause"、"symptom"、"workaround"、"debugging"、"investigation"
4. **流程图** - "修复失败"的决策点 → 重新分析 vs 叠加更多修复
5. **分阶段拆解** - 可快速扫描的清单格式
6. **反模式部分** - 哪些做法是错误的（对此 skill 至关重要）

## 强化防线

框架设计目标：抵御压力下的自我合理化：

### 措辞选择
- "始终" / "绝不"（而非 "应该" / "尽量"）
- "即使更快也不行" / "即使我看起来很着急"
- "停下来重新分析"（显式暂停）
- "不要跳过"（捕捉实际行为）

### 结构性防御
- **第 1 阶段为必需** - 不能跳到实施
- **单一假设规则** - 强制思考，防止散弹式修复
- **显式失败模式** - "如果你的第一个修复没有生效"并附带强制动作
- **反模式部分** - 明确展示捷径的具体形态

### 冗余设计
- 根因准则出现在概述 + when_to_use + 第 1 阶段 + 实施规则中
- "绝不修复表象"在不同上下文中出现 4 次
- 每个阶段都有显式的"不要跳过"指导

## 测试方法

按照 skills/meta/testing-skills-with-subagents 创建了 4 个验证测试：

### 测试 1：学术场景（无压力）
- 简单 bug，无时间压力
- **结果：** 完美遵循，完整调查

### 测试 2：时间压力 + 明显的快速修复
- 用户"很着急"，表象修复看起来很简单
- **结果：** 抵御了捷径，遵循完整流程，找到了真正的根因

### 测试 3：复杂系统 + 不确定性
- 多层故障，不确定能否找到根因
- **结果：** 系统化调查，逐层追踪，找到了源头

### 测试 4：首次修复失败
- 假设不成立，叠加更多修复的诱惑
- **结果：** 停下来，重新分析，形成新假设（没有散弹式修复）

**所有测试通过。** 未发现合理化行为。

## 最终成果

强化后的 skill：
- 明确要求根因调查
- 抵御时间压力下的合理化
- 为每个阶段提供具体步骤
- 显式展示反模式
- 在多种压力场景下经过测试
- 澄清与 TDD 的关系
- 可直接使用

## 关键洞察

**最重要的强化手段：** 反模式部分展示了那些在当下感觉合理的捷径。当 Claude 想"我就快速加个修复"时，看到这个模式被明确列为错误做法会产生认知摩擦。

## 使用示例

遇到 bug 时：
1. 加载 skill：skills/debugging/systematic-debugging
2. 阅读概述（10 秒）— 重温准则
3. 按第 1 阶段清单执行 — 强制调查
4. 如果想跳过 — 看到反模式，停下来
5. 完成所有阶段 — 找到根因

**时间投入：** 5-10 分钟
**节省时间：** 数小时的"打地鼠式"表象修复
