# 编写实施计划

## 概述

编写全面的实施计划，假设工程师对我们的代码库完全没有上下文且审美品味存疑。记录他们需要知道的一切：每个任务要改哪些文件、代码、测试、可能需要查阅的文档、如何测试。将完整计划拆分为小而可执行的任务。DRY。YAGNI。TDD。频繁提交。

假设他们是有经验的开发者，但对我们的工具链和问题域几乎一无所知。假设他们不太擅长良好的测试设计。

**启动时宣告：** "我正在使用 writing-plans skill 来创建实施计划。"

**上下文：** 如果在隔离的 worktree 中工作，它应该是在执行时通过 `zonedev:using-git-worktrees` skill 创建的。

**计划保存到：** `docs/zonedev/plans/YYYY-MM-DD-<feature-name>.md`
- （用户对计划存放位置的偏好优先于此默认值）

## 范围检查

如果设计文档涵盖了多个独立子系统，它应该在头脑风暴阶段就被拆分为子项目设计文档。如果没有，建议将其拆分为独立的计划——每个子系统一个。每个计划应该能独立产出可工作、可测试的软件。

## 文件结构

在定义任务之前，先列出哪些文件将被创建或修改，以及每个文件负责什么。这里是分解决策被锁定的地方。

- 设计边界清晰、接口定义良好的单元。每个文件应该有一个明确的职责。
- 你对能在上下文中完整把握的代码推理更准确，对职责聚焦的文件编辑更可靠。优先选择小而聚焦的文件，而非大而杂的文件。
- 一起变更的文件应该放在一起。按职责拆分，而非按技术层拆分。
- 在既有代码库中，遵循已建立的模式。如果代码库使用大文件，不要单方面重组——但如果你正在修改的文件已经变得难以维护，在计划中包含拆分是合理的。

此结构为任务分解提供依据。每个任务应该产出独立的、自成一体的变更。

## 小步执行的任务粒度

**每一步是一个动作（2-5 分钟）：**
- "编写失败的测试"——一步
- "运行它确认测试失败"——一步
- "编写最少量的代码让测试通过"——一步
- "运行测试确认通过"——一步
- "提交"——一步

## 计划文档头部

**每个计划必须以此头部开始：**

```markdown
# [功能名称] 实施计划

> **给 agentic 执行者：** 使用 zonedev:subagent-driven-development 逐任务实施此计划。步骤使用 checkbox（`- [ ]`）语法进行跟踪。

**目标：** [一句话描述要构建什么]

**架构：** [2-3 句话描述方案]

**技术栈：** [关键技术/库]

---
```

## 任务结构

````markdown
### Task N: [组件名称]

**文件：**
- 创建：`exact/path/to/file.py`
- 修改：`exact/path/to/existing.py:123-145`
- 测试：`tests/exact/path/to/test.py`

- [ ] **步骤 1：编写失败的测试**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

- [ ] **步骤 2：运行测试确认失败**

运行：`pytest tests/path/test.py::test_name -v`
预期结果：FAIL，报错 "function not defined"

- [ ] **步骤 3：编写最少量的实现**

```python
def function(input):
    return expected
```

- [ ] **步骤 4：运行测试确认通过**

运行：`pytest tests/path/test.py::test_name -v`
预期结果：PASS

- [ ] **步骤 5：提交**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
````

## 禁止占位符

每一步都必须包含工程师实际需要的内容。以下是**计划的失败**——永远不要写出这些：
- "TBD"、"TODO"、"稍后实现"、"填充细节"
- "添加适当的错误处理" / "添加验证" / "处理边界情况"
- "为上述内容编写测试"（没有实际的测试代码）
- "类似 Task N"（重复代码——工程师可能不按顺序阅读任务）
- 描述做什么但不展示如何做的步骤（代码步骤必须有代码块）
- 引用了在任何任务中都未定义的类型、函数或方法

## 要点

- 始终使用精确的文件路径
- 每个步骤中都要有完整的代码——如果一步涉及代码变更，展示代码
- 精确的命令及预期输出
- DRY、YAGNI、TDD、频繁提交

## 自审

写完完整计划后，用全新的眼光审视设计文档并对照检查计划。这是你自己执行的检查清单——不是派遣 subagent。

**1. 设计文档覆盖：** 浏览设计文档中的每个章节/需求。你能指出实现它的任务吗？列出任何遗漏。

**2. 占位符扫描：** 搜索计划中的警示信号——上述"禁止占位符"部分中的任何模式。修复它们。

**3. 类型一致性：** 你在后面任务中使用的类型、方法签名和属性名是否与前面任务中定义的一致？Task 3 中叫 `clearLayers()` 但 Task 7 中叫 `clearFullLayers()` 就是一个 bug。

发现问题就地修复。无需重新审查——修复后继续。如果发现设计文档中有需求没有对应任务，添加该任务。

## 对抗式审查

自审完成后、执行交接之前，调用 zonedev:adversarial-review skill 对计划进行对抗式审查。目的是捕获作者自身看不见的盲点——自审捕获显见问题，对抗式审查捕获隐性假设。使用多轮计划批判模式。

- **ARTIFACT：** 计划文档全文
- **CONTRACT：** 设计文档/需求

## 执行交接

保存计划后，根据计划规模自动决定执行方式：

**小型计划（≤2 个任务，且每个任务涉及 1-2 个文件）：**
- 直接在当前会话中执行，无需调用额外 skill
- 按计划步骤逐个执行，遵循 TDD 循环，每个任务完成后提交
- 完成后调用 zonedev:finishing-a-development-branch

**标准计划（3 个以上任务，或任务涉及多文件协调）：**
- **必须使用的子 SKILL：** 使用 zonedev:subagent-driven-development
- 每个任务一个全新的 subagent + 两阶段审查

**向用户报告决策：**

> "计划已完成并保存到 `<path>`。计划包含 N 个任务，[直接执行 / 使用 subagent-driven-development 执行]。"
