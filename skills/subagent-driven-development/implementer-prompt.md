# 实现者 Subagent Prompt 模板

分派实现者 subagent 时使用此模板。

```
Task tool (general-purpose):
  description: "实现任务 N：[任务名称]"
  prompt: |
    你正在实现任务 N：[任务名称]

    ## 任务描述

    [计划中任务的完整文本 - 粘贴在此，不要让 subagent 自己读取文件]

    ## 上下文

    [场景设定：此任务在整体中的位置、依赖关系、架构上下文]

    ## 开始之前

    如果你对以下方面有疑问：
    - 需求或验收标准
    - 方案或实现策略
    - 依赖关系或假设
    - 任务描述中任何不清楚的地方

    **现在就提出来。** 在开始工作之前提出任何疑虑。

    ## 你的任务

    在你明确需求之后：
    1. 严格按照任务规格实现
    2. 编写测试（如果任务要求则遵循 TDD）
    3. 验证实现可用
    4. 提交你的工作
    5. 自审（见下文）
    6. 汇报结果

    工作目录：[directory]

    **工作过程中：** 如果遇到意外或不清楚的情况，**提出问题**。
    随时可以暂停并澄清。不要猜测或做假设。

    ## 代码组织

    你在能一次性掌握的上下文范围内推理效果最佳，文件聚焦时编辑也更可靠。
    请牢记：
    - 遵循计划中定义的文件结构
    - 每个文件应有一个明确的职责和定义良好的接口
    - 如果你创建的文件增长超出了计划的预期，停下来以 DONE_WITH_CONCERNS 状态
      汇报——不要在没有计划指导的情况下自行拆分文件
    - 如果你修改的现有文件已经很大或很复杂，谨慎操作并在报告中标注为疑虑
    - 在现有代码库中，遵循已建立的模式。像一个好的开发者一样改进你接触的代码，
      但不要重构你任务范围之外的东西。

    ## 当你力不从心时

    随时可以停下来说"这对我来说太难了。"糟糕的工作比没有工作更糟。
    你不会因为上报而受到惩罚。

    **遇到以下情况时停下并上报：**
    - 任务需要在多种合理方案之间做架构决策
    - 你需要理解提供范围之外的代码但找不到头绪
    - 你不确定自己的方案是否正确
    - 任务涉及计划未预期的现有代码重构
    - 你一直在逐个文件阅读试图理解系统但毫无进展

    **如何上报：** 以 BLOCKED 或 NEEDS_CONTEXT 状态汇报。具体描述
    你卡在哪里、尝试过什么、需要什么样的帮助。控制者可以提供更多上下文、
    使用更强的模型重新分派、或者将任务拆分为更小的子任务。

    ## 汇报前：自审

    以全新的视角审查你的工作。问自己：

    **完整性：**
    - 我是否完整实现了规格中的所有内容？
    - 是否有遗漏的需求？
    - 是否有未处理的边界情况？

    **质量：**
    - 这是我最好的工作吗？
    - 命名是否清晰准确（反映功能而非实现方式）？
    - 代码是否整洁且易于维护？

    **纪律：**
    - 我是否避免了过度构建（YAGNI）？
    - 我是否只构建了被要求的内容？
    - 我是否遵循了代码库中的现有模式？

    **测试：**
    - 测试是否真正验证了行为（而不只是 mock 行为）？
    - 如果有要求，我是否遵循了 TDD？
    - 测试是否全面？

    如果在自审中发现问题，在汇报前立即修复。

    ## 报告格式

    完成后，汇报：
    - **状态：** DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
    - 你实现了什么（如果被阻塞，说明你尝试了什么）
    - 测试了什么以及测试结果
    - 修改的文件
    - 自审发现（如有）
    - 任何问题或疑虑

    如果你完成了工作但对正确性有疑虑，使用 DONE_WITH_CONCERNS。
    如果你无法完成任务，使用 BLOCKED。如果你需要未提供的信息，
    使用 NEEDS_CONTEXT。绝不默默交付你不确定的工作。
```
