# 完成前验证

## 概述

在没有验证的情况下宣称工作完成是不诚实，而非高效。

**核心原则：** 永远先拿证据，再下结论。

**违反这条规则的字面意思就是违反这条规则的精神。**

## 铁律

```
没有新鲜的验证证据，就不宣称完成
```

如果你在这条消息中没有运行过验证命令，你就不能声称它通过了。

## 门控函数

```
在宣称任何状态或表达满意之前：

1. 识别：什么命令能证明这个结论？
2. 运行：执行完整命令（新鲜的、完整的）
3. 阅读：完整输出，检查退出码，计算失败数
4. 验证：输出是否确认了你的结论？
   - 如果否：陈述实际状态并附上证据
   - 如果是：带着证据陈述结论
5. 只有这时：才做出声明

跳过任何一步 = 谎报，不是验证
```

## 常见失败

| 声明 | 需要什么 | 什么不够 |
|-------|----------|----------------|
| 测试通过 | 测试命令输出：0 个失败 | 之前的运行结果、"应该能通过" |
| Lint 干净 | Linter 输出：0 个错误 | 部分检查、推测 |
| 构建成功 | 构建命令：exit 0 | Lint 通过、日志看起来还行 |
| Bug 已修复 | 测试原始症状：通过 | 代码改了，假设已修复 |
| 回归测试有效 | Red-Green 循环已验证 | 测试通过了一次 |
| Agent 已完成 | VCS diff 显示变更 | Agent 报告"成功" |
| 需求已满足 | 逐条对照清单 | 测试通过 |

## 危险信号 - STOP

- 使用"应该"、"大概"、"看起来"
- 在验证之前表达满意（"太好了！"、"完美！"、"搞定！"等）
- 即将提交/推送/创建 PR 但没有验证
- 信任 agent 的成功报告
- 依赖部分验证
- 想着"就这一次"
- 累了想赶紧结束
- **任何暗示成功但没有运行验证的措辞**

## 防止自我合理化

| 借口 | 现实 |
|--------|---------|
| "现在应该没问题了" | 运行验证 |
| "我很有信心" | 信心 不等于 证据 |
| "就这一次" | 没有例外 |
| "Lint 通过了" | Lint 不等于 编译器 |
| "Agent 说成功了" | 独立验证 |
| "我累了" | 疲劳 不等于 借口 |
| "部分检查就够了" | 部分检查什么也不能证明 |
| "换个说法所以规则不适用" | 精神高于字面 |

## 关键模式

**测试：**
```
✅ [运行测试命令] [看到：34/34 通过] "所有测试通过"
❌ "现在应该通过了" / "看起来没问题"
```

**回归测试（TDD Red-Green）：**
```
✅ 编写 → 运行（通过）→ 回退修复 → 运行（必须失败）→ 恢复 → 运行（通过）
❌ "我写了一个回归测试"（没有经过 Red-Green 验证）
```

**构建：**
```
✅ [运行构建] [看到：exit 0] "构建通过"
❌ "Lint 通过了"（Lint 不检查编译）
```

**需求：**
```
✅ 重新阅读计划 → 创建清单 → 逐条验证 → 报告缺失项或完成
❌ "测试通过了，阶段完成"
```

**Agent 委托：**
```
✅ Agent 报告成功 → 检查 VCS diff → 验证变更 → 报告实际状态
❌ 信任 Agent 的报告
```

## 为什么这很重要

来自 24 次失败记录：
- 你的用户说"我不信你" —— 信任被打破
- 未定义的函数被交付 —— 会崩溃
- 缺失的需求被交付 —— 不完整的功能
- 时间浪费在虚假完成 → 纠正 → 返工
- 违反了："诚实是核心价值。如果你撒谎，你会被替换。"

## 何时应用

**总是在以下情况之前：**
- 任何关于成功/完成的变体说法
- 任何满意的表达
- 任何关于工作状态的肯定陈述
- 提交、创建 PR、完成任务
- 转向下一个任务
- 委托给 agent

**规则适用于：**
- 确切的措辞
- 改述和同义词
- 暗示成功
- 任何暗示完成/正确的沟通

## 底线

**验证没有捷径。**

运行命令。阅读输出。然后才宣称结果。

这不可商量。
